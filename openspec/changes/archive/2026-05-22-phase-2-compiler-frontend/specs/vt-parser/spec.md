## ADDED Requirements

### Requirement: Parser transforms token stream into AST
The `vt-parser` crate SHALL accept a sequence of `vt_lexer::Token` values and produce a valid `vt_core::Program` AST when the input conforms to the VisioTalk EBNF grammar defined in `docs/grammar.md`. The parser SHALL be an LR(1) parser generated by LALRPOP from a `.lalrpop` grammar file.

#### Scenario: Parse complete example with mode and multiple rules
- **WHEN** parsing the canonical example `.vt` from `docs/PRD.md` section 5.1 (mode `navegacion_urbana` + three `DETECTAR` rules with distance, alert, and priority attributes)
- **THEN** the parser SHALL produce a `Program` with `mode` field containing `ModeDecl { name: "navegacion_urbana" }` and `rules` containing exactly three `DetectRule` nodes in order

#### Scenario: Parse minimal program without mode declaration
- **WHEN** parsing a `.vt` file containing only `DETECTAR persona { distancia: < 2.0m alerta: "Alerta" prioridad: 1 }` with no `MODO` declaration
- **THEN** the parser SHALL produce a `Program` with `mode: None` and `rules` containing one `DetectRule`

#### Scenario: Parse rule with all three attribute types
- **WHEN** parsing a `DETECTAR` block containing `distancia:`, `alerta:`, and `prioridad:` attributes in any order
- **THEN** the parser SHALL produce a `DetectRule` with three `Attribute` variants (`Distance`, `Alert`, `Priority`) in declaration order

### Requirement: Parser rejects invalid syntax
The parser SHALL return structured errors when the input does not conform to the grammar. Errors SHALL include the unexpected token, its location (line/column), and the set of expected tokens.

#### Scenario: Reject missing closing brace
- **WHEN** parsing `DETECTAR persona { distancia: < 2.0m` without a closing `}`
- **THEN** the parser SHALL return a `ParseError` indicating unexpected end of input and expecting `}`

#### Scenario: Reject unknown attribute key
- **WHEN** parsing a rule block containing `altura: 1.0m` where `altura` is not a recognized attribute key
- **THEN** the parser SHALL return a `ParseError` indicating `altura` is not a valid attribute (expected one of `distancia`, `alerta`, `prioridad`)

#### Scenario: Reject missing required attribute values
- **WHEN** parsing `DETECTAR persona { distancia: < }` with a comparator but no numeric value
- **THEN** the parser SHALL return a `ParseError` indicating a number literal was expected

### Requirement: Parser handles whitespace and comments
The parser SHALL correctly parse `.vt` files containing arbitrary whitespace (spaces, tabs, newlines) and single-line comments (`//`). Comments SHALL be ignored by the lexer and not affect AST construction.

#### Scenario: Parse file with mixed whitespace and comments
- **WHEN** parsing a `.vt` file containing extra blank lines, indentation, and `//` comments between rules
- **THEN** the produced AST SHALL be identical to parsing the same file without whitespace or comments

### Requirement: Parser accepts all distance comparator and unit combinations
The parser SHALL accept any valid combination of comparator (`<`, `>`, `<=`, `>=`, `==`) and unit (`m`, `cm`, `ft`, `px`) in a `distancia:` expression.

#### Scenario: Parse all five comparators
- **WHEN** parsing `distancia: < 1.0m`, `distancia: > 1.0m`, `distancia: <= 1.0m`, `distancia: >= 1.0m`, and `distancia: == 1.0m` in separate rules
- **THEN** each SHALL produce the corresponding `Comparator` variant in the `DistanceExpr`

#### Scenario: Parse all four distance units
- **WHEN** parsing rules with `distancia: < 1.0m`, `distancia: < 100cm`, `distancia: < 3.0ft`, and `distancia: < 500px`
- **THEN** each SHALL produce the corresponding `Unit` variant in the `DistanceExpr`

### Requirement: Parser accepts integer and floating-point numeric literals
The parser SHALL accept both integer and floating-point numeric values in `distancia:` expressions and `prioridad:` attributes.

#### Scenario: Parse integer and float values
- **WHEN** parsing `distancia: < 2m` (integer) and `distancia: < 2.5m` (float)
- **THEN** `DistanceExpr.value` SHALL be `2.0` and `2.5` respectively

### Requirement: AST output is deterministic for equivalent inputs
Parsing the same `.vt` source SHALL produce bitwise-identical `Program` values regardless of whitespace or comment variations. This property enables reliable snapshot testing.

#### Scenario: Snapshot stability
- **WHEN** parsing the same `.vt` file with `insta::assert_debug_snapshot!` across multiple test runs
- **THEN** the snapshot SHALL match identically without requiring update unless the grammar or AST types change
